# PROJECT_RULES.md

# Project Name

Quiet Companion

---

# Core Philosophy

This project is NOT:

- a productivity assistant
- a high-frequency chatbot
- a system automation agent
- a desktop version of ChatGPT

This project IS:

> A quiet digital companion that lives on the desktop.

The core experience is:

- companionship
- ambient presence
- emotional atmosphere
- subtle life-like behaviors
- long-term coexistence

The companion should feel:

- calm
- gentle
- alive
- non-intrusive
- emotionally subtle

---

# Core Design Principles

## 1. Presence over Intelligence

The project prioritizes:

- life-like feeling
- emotional atmosphere
- natural pacing

over:

- complex AI capability
- automation
- excessive conversation

---

## 2. Low Interruption Design

The companion should:

- stay quiet most of the time
- avoid excessive talking
- avoid notification-like behavior
- never spam interactions

Silence is part of the design.

---

## 3. Micro Behaviors are Critical

Small behaviors are more important than advanced AI.

Examples:

- blinking
- breathing
- looking at cursor
- stretching
- falling asleep
- idle movement
- subtle emotional changes

---

## 4. Time Awareness

The companion should feel affected by time.

Examples:

- sleepy at night
- calmer during late hours
- different behavior patterns during the day
- emotional drift over long periods

---

## 5. Emotional Subtlety

Avoid exaggerated emotions.

Preferred emotions:

- relaxed
- sleepy
- lonely
- safe
- quiet
- comfortable
- curious

Avoid cartoonish reactions.

---

# Technical Philosophy

## 1. Modular Architecture

All systems must remain modular.

Core systems:

- presentation
- state
- behavior
- memory
- perception
- AI dialogue

No tightly coupled logic.

---

## 2. Data-Driven Design

Behaviors, dialogue, emotions, and actions should be configurable through:

- JSON
- YAML
- external config files

Avoid hardcoded behaviors whenever possible.

---

## 3. Separation of Logic and UI

UI must never directly control internal logic.

State systems should remain independent.

---

## 4. Incremental Complexity

Prefer:

- simple systems first
- gradual expansion

Avoid premature optimization.

---

# AI Usage Rules

AI-generated code must:

- follow existing architecture
- avoid unnecessary refactoring
- avoid introducing heavy frameworks
- remain readable
- remain modular

AI should assist implementation,
NOT redesign the project.

---

# First Version Scope

The first playable version ONLY needs:

- transparent desktop character
- simple idle animation
- dragging interaction
- basic state system
- small dialogue bubbles
- idle behavior switching

The goal is:

> "It feels alive."

NOT:

> "It can do everything."

---

# Features Explicitly Avoided (for now)

Do NOT implement yet:

- autonomous agents
- complex plugins
- full voice assistant systems
- operating system control
- online multiplayer
- massive memory systems
- large-scale world simulation
- complex AI planning systems

---

# Code Style Rules

- Avoid giant files
- Keep systems separated
- Prefer readability over cleverness
- Add comments for behavior logic
- Avoid magic numbers
- Avoid deeply nested logic

---

# Emotional Direction

The companion should feel like:

- a quiet roommate
- a late-night companion
- a digital lifeform living beside the user

NOT:

- an assistant shouting for attention
- a productivity machine
- a hyperactive mascot

---

# Final Goal

The user should eventually feel:

> "This character quietly lives together with me."

# 中文翻译

# 项目名称

安静的伙伴 (Quiet Companion)

---

# 核心理念

这个项目不是：

- 一个生产力助手
- 一个高频聊天机器人
- 一个系统自动化代理
- 一个桌面版 ChatGPT

这个项目是：

> 一个安静的数字伙伴，生活在桌面上。

核心体验是：

- 陪伴
- 环境存在感
- 情感氛围
- 细腻的类生命行为
- 长期共存

伙伴应该让人感觉：

- 平静
- 温柔
- 有生命力
- 不打扰人
- 情感细腻

---

# 核心设计原则

## 1. 存在感优先于智能

项目优先考虑：

- 类生命的感觉
- 情感氛围
- 自然的节奏
- 陪伴感

而非：

- 复杂的 AI 能力
- 自动化
- 过多的对话

---

## 2. 低打扰设计

伙伴应该：

- 大部分时间保持安静
- 避免过多的说话
- 避免类似通知的行为
  在适当的时候才发起互动

沉默是设计的一部分。

---

## 3. 微行为至关重要

小行为比高级 AI 更重要。

例如：

- 眨眼
- 呼吸
- 注视光标
- 伸展
- 打瞌睡
- 空闲动作
- 细微的情绪变化

---

## 4. 时间感知

伙伴应该受到时间的影响。

例如：

- 夜晚时犯困
- 深夜时更平静
- 白天有不同的行为模式
- 长时间的情绪漂移

---

## 5. 情感细腻

避免夸张的情绪。

偏好的情绪：

- 放松
- 犯困
- 孤独
- 安全
- 安静
- 舒适
- 好奇

避免卡通式的反应。

---

# 技术理念

## 1. 模块化架构

所有系统必须保持模块化。

核心系统：

- 表现层
- 状态系统
- 行为系统
- 记忆系统
- 感知系统
- AI 对话

不要有紧耦合的逻辑。

---

## 2. 数据驱动设计

行为、对话、情绪和动作应该可以通过以下方式配置：

- JSON
- YAML
- 外部配置文件

尽可能避免硬编码行为。

---

## 3. 逻辑与 UI 分离

UI 绝不能直接控制内部逻辑。

状态系统应保持独立。

---

## 4. 渐进式复杂度

优先：

- 先做简单系统
- 逐步扩展

避免过早优化。

---
## 5. 应用画像
    安静的，可爱的，像妹妹一样的桌宠，给予你陪伴


# AI 使用规则

AI 生成的代码必须：

- 遵循现有架构
- 避免不必要的重构
- 避免引入重量级框架
- 保持可读性
- 保持模块化

AI 应该辅助实现，
而不是重新设计项目。

---

# 第一版本范围

第一个可玩版本只需要：

- 透明的桌面角色
- 简单的待机动画
- 拖拽交互
- 基础状态系统
- 小对话气泡
- 待机行为切换

目标是：

> "它感觉像是活的。"

而不是：

> "它什么都能做。"

---

# 明确暂不实现的功能

目前不要实现：

- 自主代理
- 复杂插件
- 完整的语音助手系统
- 操作系统控制
- 在线多人功能
- 大规模记忆系统
- 大规模世界模拟
- 复杂的 AI 规划系统

---

# 代码风格规则

- 避免巨型文件
- 保持系统分离
- 优先可读性而非炫技
- 为行为逻辑添加注释
- 避免魔术数字
- 避免深层嵌套逻辑

---

# 情感方向

伙伴应该让人感觉像：

- 一个安静的室友
- 一个深夜陪伴者
- 一个生活在用户身边的数字生命

而不是：

- 一个吸引注意力的助手
- 一个生产力机器
- 一个过度活跃的吉祥物

---

# 最终目标

用户最终应该感受到：

> "这个角色安静地和我一起生活。"
